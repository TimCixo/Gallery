Коли надходить запит на коміт, спочатку переглянь зміни та стиль оформлення попередніх комітів. Не формуй повідомлення навмання.

Перед комітом звір вимоги з `AGENTS.md`. Для цього репозиторію документацію треба оновлювати за потреби в одному з файлів:

- `README.md`
- `GalleryApp/frontend/README.md`
- `GalleryApp/backend/README.md`

## Workflow

1. Перевір `git status`.
2. Переглянь `git diff` або staged diff для змінених файлів.
3. Переглянь кілька останніх комітів, щоб зберегти стиль репозиторію.
4. Переконайся, що потрібна документація оновлена, якщо зміни зачіпають поведінку, workflow, setup або структуру.
5. Згрупуй зміни за логічними результатами. Якщо зміни незалежні, готуй окремі коміти.
6. Вибери marker за основним типом зміни.
7. Підготуй коротке повідомлення в імперативному стилі.
8. Лише після цього виконуй `git add` і `git commit`.

## Format

```text
<marker>: <description>
```

За потреби додавай область:

```text
feat(media): add quick tagging workflow
fix(ui): correct modal footer alignment
```

## Markers

- `feat` для нової функціональності
- `fix` для виправлення багів
- `chore` для технічних змін без user-facing ефекту
- `refactor` для реструктуризації без зміни поведінки
- `style` для форматування та не-функціональних правок
- `docs` для документації
- `test` для тестів
- `perf` для продуктивності
- `ci` для CI/CD
- `build` для змін збірки або конфігурації збірки
- `revert` для відкату

## Rules

- Після marker став двокрапку.
- Опис пиши в імперативному стилі.
- Перше слово після marker має бути з малої літери, крім власних назв.
- Тримай повідомлення коротким і конкретним.
- Не змішуй в одному коміті незалежні зміни, якщо їх можна безпечно розділити.

## Do Not

- Не коміть без перегляду diff.
- Не пропускай перевірку стилю попередніх комітів.
- Не коміть функціональні зміни без оновлення документації, якщо вона стала неактуальною.
- Не використовуй `feat` як універсальний marker для будь-чого.
