Назва навички лишилась історичною, але в цьому репозиторії вона застосовується до веб-архітектури, а не до Unity. Працюй прагматично: шукай не абстрактно "ідеальну архітектуру", а місця, де код стане крихким, дубльованим або заплутаним.

## Workflow

1. Визнач межі підсистеми:
   - які компоненти, hooks, utils або backend-сервіси беруть участь;
   - хто володіє станом;
   - де проходять межі між UI, state, data fetching і domain logic.
2. Перевір ключові ризики:
   - duplicated logic;
   - tight coupling між feature-модулями;
   - змішування presentation і data logic;
   - неочевидний ownership стану;
   - helper-утиліти, що перетворилися на god-object.
3. Оціни, чи вже є в репозиторії патерн, який треба продовжити, а не винаходити новий.
4. Сформуй короткий review:
   - що добре як є;
   - що реально проблемне;
   - що варто спростити або рознести;
   - які компроміси це дає.

## Output Format

Відповідь будуй у порядку:

1. `Findings`
2. `Recommendations`
3. `Tradeoffs`

## Rules

- Віддавай перевагу мінімальним змінам структури.
- Не пропонуй нові шари абстракції без явної вигоди.
- Для frontend орієнтуйся на `app/`, `features/`, `shared/`, hooks, utils і test coverage.
- Для backend орієнтуйся на фактичні модулі, API boundaries, services і data contracts.
- Якщо поточна структура вже достатньо добра, прямо так і скажи.

## Do Not

- Не тягни в repo Unity-специфічні поради.
- Не просувай DI container, service locator, event bus чи іншу важку інфраструктуру без явної потреби.
- Не маскуй смакові вподобання під об'єктивні проблеми.
- Не переписуй код у межах цієї навички, якщо користувач просить саме review.
