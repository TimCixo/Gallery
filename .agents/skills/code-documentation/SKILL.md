Пиши документацію лише в коді. Для Gallery repo це означає:

- `/** ... */` або JSDoc у `GalleryApp/frontend/src/**`
- XML documentation comments у C# коді backend, якщо вони там доречні
- короткі inline-коментарі лише біля справді неочевидної логіки

Не оновлюй `README.md`, `GalleryApp/frontend/README.md` чи `GalleryApp/backend/README.md` у межах цієї навички.

## Workflow

1. Прочитай реалізацію і зрозумій реальну поведінку символу або блоку коду.
2. Перевір, чи в коді вже є усталений стиль документації, і продовжуй саме його.
3. Документуй лише ті місця, де це реально зменшує час на читання:
   - публічні API;
   - неочевидні контракти параметрів і return value;
   - побічні ефекти;
   - домовленості між frontend/backend;
   - нетривіальні алгоритми або edge cases.
4. Якщо коментар дублює очевидний код, не додавай його.
5. Після змін перевір, що документація не суперечить сигнатурам і фактичній поведінці.

## Rules

- Пиши коротко і по факту.
- Документуй контракт, а не переказуй тіло функції.
- Для React/JS коду пояснюй props, side effects, data shape або призначення helper-функцій, якщо це неочевидно.
- Для backend-коду пояснюй вхідні дані, винятки, значення що повертаються і важливі обмеження.
- Якщо в коді вже вистачає виразних назв і структури, не засмічуй файл коментарями.

## Do Not

- Не змінюй документацію проєкту поза кодом.
- Не вигадуй поведінку, якої немає в реалізації.
- Не додавай декоративні або шумні коментарі.
- Не документуй приватні дрібні helper-и автоматично; роби це лише коли без коментаря код реально важче підтримувати.
