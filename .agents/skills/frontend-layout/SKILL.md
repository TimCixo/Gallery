Орієнтуйся на вже наявні патерни в:

- `GalleryApp/frontend/src/App.css`
- `GalleryApp/frontend/src/features/**`
- спільні класи кнопок, модалок, toolbar і pagination

Спочатку шукай, чи є в проєкті готовий клас, компонент або патерн, який можна перевикористати. Нові стилі додавай лише коли це справді потрібно.

## Workflow

1. Знайди існуючу реалізацію схожого UI-елемента.
2. Перевір, чи можна перевикористати наявні CSS-класи, markup-структуру або компонент.
3. Виправляй layout через мінімальні зміни:
   - вирівнювання;
   - ширини;
   - spacing;
   - overflow;
   - sizing;
   - visual states.
4. Якщо потрібна нова обгортка або клас, роби його вузьким і локальним за відповідальністю.
5. Після змін обов'язково перевір, що UI не ламає суміжні стани, і онови/додай unit tests, якщо ці стани вже тестуються.

## Rules

- Не дублюй існуючі button/modal/pagination стилі без потреби.
- Віддавай перевагу спільним класам замість одноразових override-костилів.
- Не змінюй логіку компонента, якщо проблему можна вирішити layout-правкою.
- Якщо layout залежить від умовного рендерингу, перевір усі ключові стани: empty, active, disabled, filtered.
- Для нового UI дотримуйся візуальної мови проєкту, а не принось нову систему стилів.

## Do Not

- Не додай великі CSS-блоки там, де вистачає перевикористання наявного класу.
- Не ламай існуючі toolbar або modal patterns заради локального виправлення.
- Не змінюй user-facing behavior, якщо запит стосується лише presentation/layout.
